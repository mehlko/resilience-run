ğŸŒ Core Goals of this project

  • Goal: Increase the system resilience of critical infrastructure during design and runtime.
  • Subgoal1: Provide a model-driven method to define resilience requirements.
  • Subgoal2: Provide an automated method to measure compliance of models with their resilience requirements.
  • Subgoal3: Provide a method to define, select and apply mechanisms to increase compliance with resilience requirements.
  • Subgoal4: Provide ready-to-use tool prototypes that can be used to evaluate our approach on real-world use-cases.

ğŸ“ Resilience Modeling

  • RQ1: How can resilience requirements be represented in system models?
  • Contribution1.1: Method to annotate the models and instances of system models with resilience requirements.
  • Contribution1.2: Formal and well-defined documentation (e.g. feature model and glossary) of selected key resilience requirements and mechanisms that is understandable to system designers and operators.
  • RQ2: How can the compliance of system models with resilience requirements be automatically assessed?
  • Contribution2.1: Design-time method to statically analyze the resilience of a system based on the annotated system models.
  • Contribution2.2: Runtime time method to dynamically analyze the resilience of a system based on the annotated system models and their instances.
  • Contribution2.3: Method to (also visually) link models and instances of processes, their requirements and assets.
  • RQ3: How can the mechanism be defined, selected and applied to the system models to increase compliance with resilience requirements?
  • Contribution3.1 Template for defining detection, response and recovery mechanisms.
  • Contribution3.2 Method to select and apply detection, response and recovery mechanisms based on the system analysis.

Use

  • Use modeling to prioritize which areas of a system need to be made more resilient.
  • Use modeling techniques to identify potential sources of system disruptions and their impact on system resilience.

ğŸ“ˆ Static Resilience Analysis

    Use

    • Use automated static analysis tools to to identify potential system vulnerabilities.
    • Use static resilience analysis in conjunction with other analyses, such as security analysis, to assess system resilience.
    • Use static analysis results to prioritize which areas of a system need to be made more resilient.
    • Conduct regular automated static analysis of systems and use the results to improve resilience over time.
    • Use static analysis results from one system to help assess the resilience of similar systems.
    • Share static analysis results among trusted parties in order to improve overall system resilience.
    • Develop custom static analysis rules tailored specifically for assessing system resilience.
    • Integrate static analysis into DevOps processes so that potential issues are identified and addressed early on during development
